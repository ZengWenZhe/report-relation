## 背景

我们正在设计一套新的工作协同内容处理系统（Processing Core），位于工作协同系统与 BP 系统之间，实现汇报内容的标准化、结构化输出，供 BP 等业务系统消费。

经过多轮讨论，Processing Core 的整体架构已初步确认，现在需要你从**工作协同系统侧**提供以下信息和支持：

## 需要你确认/提供的 3 个事项

### 事项 1：MQ 事件通道可行性确认（Critical）

Processing Core 需要监听工作协同系统的以下 MQ 事件：

| 事件类型 | 说明 | 是否必须 |
|---------|------|---------|
| `关联变更` | 汇报被打上/取消 BP 标签时触发，需含标签类型字段 | 必须 |
| `内容变更` | 汇报正文内容发生变更时触发 | 必须 |
| `回复新增` | 汇报下新增评论/回复时触发 | 必须 |
| `附件变更` | 汇报附件发生变更时触发 | 建议 |

**请确认：**
1. 工作协同系统是否支持以上 MQ 事件推送？
2. 各事件的字段格式是什么（特别是 `关联变更` 事件是否包含标签类型字段）？
3. 如果不支持，能否提供替代方案？

---

### 事项 2：阶段一完整方案（文件包结构）

Processing Core 的阶段一（汇报文件包生成）需要以下信息：

**汇报文件包结构（草案）：**
```
report_{reportId}/
├── index.md          # 主文件
└── attachments/      # 附件目录
```

**`index.md` 需要包含：**
- 标题、创建时间、更新时间、提交人 ID
- 正文（去 HTML，保留原文结构）
- 评论/回复（按时间顺序，含评论人、时间、内容）
- 附件清单（附件名称 + 链接）

**请确认/提供：**
1. `index.md` 的具体格式规范
2. 附件链接的格式（绝对 URL 还是相对路径？）
3. 附件摘要（100-200 字）由谁提取？工作协同侧还是 Processing Core 侧？

---

### 事项 3：AI 汇报类型处理

工作协同中存在两类 AI 汇报，需要你提供判断规则：

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| 搬运型 AI 汇报 | 业务驱动，AI 对已有汇报整理汇总，无新信息 | 忽略，不纳入处理 |
| 业务新增型 AI 汇报 | 业务系统调用发送接口产生，是全新的工作记录 | 纳入，正常处理 |

**请确认/提供：**
1. 这两类 AI 汇报在系统层面如何区分？
2. 判断结果通过什么字段/机制传递给 Processing Core？
